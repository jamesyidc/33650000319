🐛 问题5：XRP等大面值合约开仓金额严重超额（提交: 22be347）

  • 现象：XRP每次开仓金额都不对，目标5U保证金（50U合约价值），实际开出14.13 USDT保证金（140U合约价值），超额2.8倍
  • 受影响币种：XRP(100/张)、DOGE(1000/张)、ADA(100/张)、TRX(1000/张) - 所有大面值合约
  • 根因分析：
    • ① XRP真实合约规格：ctVal=100 XRP/张，每张≈140 USDT（价格1.40）
    • ② 目标开仓50 USDT → 需要 50/140 = 0.36张
    • ③ 致命Bug：代码强制 max(1, round(0.36)) = 1张
    • ④ 实际开仓：1张 = 140 USDT合约价值 = 14 USDT保证金 ❌
  • 错误代码：contracts_count = max(1, round(...)) # 强制至少1张
  • 误解根源：OKX支持小数张数（minSz=0.01），不需要强制取整为1张

✅ 解决方案：

  • 移除强制逻辑：删除 max(1, ...)，改为 round(contracts_count, 2)（保留2位小数）
  • 支持小数张数：0.36张 → 实际开仓50.4 USDT ✅（误差仅0.8%）
  • 最小下单量：如果计算结果 < 0.01张，则使用0.01张（而不是1张）
  • 修复后效果：
    • 目标50 USDT → 计算0.36张 → 实际开仓：0.36×140 = 50.4 USDT ✅
    • 目标5 USDT → 计算0.036张 → 实际开仓：0.04×140 = 5.6 USDT ✅
    • 超额倍数：从2.8倍降低到1.01倍（±1%误差）

⚠️ 危险性说明：

  • XRP：1张=100 XRP，强制1张 → 超额2.8倍
  • DOGE：1张=1000 DOGE，强制1张 → 超额10倍+
  • 风险等级：🔴 致命级（可能导致爆仓或重大亏损）

📊 positionSize 字段说明（v2.6新增）

  • 作用：每个币种的开仓金额占总权益的百分比
  • 默认值：1.5（即1.5%）
  • 计算公式：开仓金额 = min(总权益 × positionSize%, 5 USDT)
  • 推荐配置：🟢 保守 0.5% | 🟡 稳健 1.5% | 🔴 激进 3.0%
  • 示例：总权益198.64 USDT，positionSize=1.5 → 每币2.98 USDT，8币总计23.84 USDT
  • 修改方式：直接编辑JSON文件，保存后下次触发时生效

🛡️ 十、防重复下单机制（三层防护）

系统采用三层防护机制，确保即使多设备/多浏览器同时访问，也只执行一次开单操作。

10.1 第一层：JSONL文件锁（数据层）- 最可靠的防护

  • 🆕 BTC价格触发策略（v2.5独立）：每个账号、每个策略类型独立的执行许可文件：
    • data/okx_auto_strategy/account_{account_id}_btc_top_performers_execution.jsonl
    • data/okx_auto_strategy/account_{account_id}_btc_bottom_performers_execution.jsonl
  • 🆕 上涨占比0策略：每个账号、每个策略独立的执行许可文件：
    • data/okx_auto_strategy/account_{account_id}_upratio0_top8_execution.jsonl
    • data/okx_auto_strategy/account_{account_id}_upratio0_bottom8_execution.jsonl
  • 记录格式：{"allowed": true/false, "reason": "...", "timestamp": "...", "account_id": "...", "strategy_type": "..."}
  • 执行前通过API检查文件最后一条记录的 allowed 状态
  • 执行后立即通过API写入 allowed: false，阻止重复触发
  • ✅ 跨设备、跨浏览器、跨会话有效 - 这是唯一真相源

10.2 第二层：冷却期（时间层）

  • 5分钟强制冷却期，防止短时间内重复执行
  • 检查 lastExecutedTime，距离上次执行不足5分钟则拒绝

10.3 第三层：执行锁（进程层）

  • 全局变量 strategyExecuting 标记执行中状态
  • 同一浏览器页面内防止并发执行

10.4 工作流程

  1. 用户开启开关 → POST /api/okx-trading/set-allowed/{account_id} 写入 allowed: true
  2. 触发条件满足 → GET /api/okx-trading/check-allowed/{account_id} 检查JSONL
  3. 检查冷却期（5分钟）
  4. 设置前端执行锁
  5. 执行下单操作
  6. 下单成功 → POST /api/okx-trading/set-allowed/{account_id} 写入 allowed: false
  7. 释放前端执行锁

⚠️ 重要：即使多个浏览器/设备同时打开，JSONL文件锁确保只有第一个触发的请求能够执行。这是最可靠的防护层。

📚 详细功能说明 ▼

🎯 工作原理

  • 监控频率：每60秒检查一次27个主要币种的24小时价格变化
  • 触发计算：将27个币种的涨跌幅相加得到总值
  • 做空条件：当总值 ≥ 设定阈值（如50%）时触发做空策略
  • 做多条件：当总值 ≤ 设定阈值（如-30%）时触发做多策略

🔒 防重复触发机制

  • 触发一次后，allow_trigger自动设为false
  • 需要人工判断市场情况后，手动点击"重置"按钮
  • 重置后权限恢复，可再次触发
  • 目的：避免极端行情下连续触发导致损失放大

🔑 执行反手的3个必要条件

条件 说明 如何设置
1️⃣ 启用开关 打开"启用反手开单"开关
2️⃣ 触发权限 首次启用自动为true；触发后需点击"重置"

⚠️ 重要：必须同时满足以上3个条件才会执行反手！缺少任何一个条件都会跳过。

🚫 不会执行反手的情况

  • ⏸️ 未启用：开关关闭 → 日志提示"配置未启用"
  • 🔒 已触发：触发后自动禁用 → 日志提示"触发权限已禁用，需要手动重置"
  • ⚠️ 无策略：未选择反手策略 → 日志提示"未设置目标策略"
  • ⚪ 无配置：首次使用未创建配置 → 访问前端页面自动创建

📡 通知方式

  • Telegram通知：实时推送到手机，包含止损信息、反手策略、执行结果
  • 前端弹窗：右上角精美弹窗，10秒倒计时自动关闭

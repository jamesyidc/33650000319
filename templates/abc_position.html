🎯 系统概述

ABC开仓系统是一个基于OKX交易所的多账户、多仓位智能监控与管理系统。系统支持4个独立账户（主账户、POIT、fangfang12、dadanini），每个账户最多可管理ABC三个仓位，根据持仓成本动态计算风险等级并显示可视化曲线。

核心功能

  • 🔄 实时持仓监控：每60秒自动从OKX API获取最新持仓数据
  • 📊 成本阈值管理：基于USDT成本判断A/B/C仓位状态（绿/黄/红/橙色）
  • 📈 历史曲线展示：可视化显示各账户盈亏百分比变化趋势
  • ⚙️ 独立阈值配置：每个账户可单独设置ABC三个成本阈值
  • 🔐 开单权限控制：支持启用/禁用交易，限制做多/做空方向
  • 💾 JSONL数据存储：所有数据保存为JSONL格式，易于分析

⚙️ 系统运行逻辑

1. 数据采集流程

OKX API
  ↓ (每60秒)
abc_position_tracker.py
  ↓ 解析持仓数据
计算成本、盈亏、持仓数
  ↓ 读取阈值配置
判断颜色状态 (green/yellow/red/orange)
  ↓ 保存状态
abc_position_state.json
  ↓ 追加历史记录
abc_position_[account]_[date].jsonl

2. 颜色判断逻辑

成本范围 颜色 状态 含义
0 USDT ⚪ none 无持仓 当前无仓位
≤ A阈值 (例如≤20) 🟢 green 仅A仓 低成本，正常范围
A < 成本 ≤ B (例如20-40) 🟡 yellow AB仓 中等成本，需注意
B < 成本 ≤ C (例如40-55) 🔴 red ABC仓 高成本，风险警告
> C阈值 (例如>55) 🟠 orange 超限 成本过高，严重警告

3. 前端更新流程

页面加载
  ↓
loadState() → /api/current-state
  ↓ 获取当前状态
updateUI() → 更新账户卡片、颜色指示器
  ↓
loadTodayChart() → /api/history?date=YYYYMMDD
  ↓ 获取历史数据
updateChart() → 绘制ECharts曲线图
  ↓ (每60秒循环)

数据流转时序

时间 组件 操作 结果
T+0s tracker 调用OKX API获取4个账户持仓 获得原始持仓数据
T+1s tracker 解析数据，计算total_cost、unrealized_pnl、pnl_pct 获得计算指标
T+2s tracker 读取abc_position_settings.json获取ABC阈值 获得阈值配置
T+3s tracker 调用get_position_color()判断颜色状态 返回green/yellow/red/orange/none
T+4s tracker 更新abc_position_state.json（覆盖写入） 当前状态已保存
T+5s tracker 追加记录到abc_position_*_YYYYMMDD.jsonl（4个文件） 历史数据已记录
T+60s tracker 等待60秒后重新循环 进入下一轮采集
任意时刻 浏览器 请求/api/current-state 读取state.json返回最新数据
任意时刻 浏览器 请求/api/history?date=20260314 读取指定日期JSONL返回历史曲线

核心算法：颜色判断逻辑

# abc_position_tracker.py - get_position_color()

def get_position_color(account_id, total_cost):
    """
    根据账户成本（USDT）判断颜色状态
    
    参数:
        account_id: 账户ID（'A', 'B', 'C', 'D'）
        total_cost: 持仓总成本（USDT）
    
    返回:
        颜色字符串：'none' | 'green' | 'yellow' | 'red' | 'orange'
    """
    # 读取阈值配置
    settings = read_json('data/abc_position/abc_position_settings.json')
    thresholds = settings['accounts'][account_id]['thresholds']
    
    A = thresholds['A']  # 例如 20 USDT
    B = thresholds['B']  # 例如 40 USDT
    C = thresholds['C']  # 例如 55 USDT
    
    # 判断逻辑
    if total_cost == 0:
        return 'none'      # 无持仓
    elif total_cost <= A:
        return 'green'     # 仅A仓（成本≤20）
    elif total_cost <= B:
        return 'yellow'    # AB仓（20<成本≤40）
    elif total_cost <= C:
        return 'red'       # ABC仓（40<成本≤55）
    else:
        return 'orange'    # 超限（成本>55）

性能优化措施

  • 分布式采集：4个账户并发请求OKX API，减少总耗时
  • JSON缓存：state.json作为缓存，避免频繁读取JSONL文件
  • JSONL追加写入：历史数据使用追加模式，避免全文件读写
  • 前端60秒轮询：与后端60秒采集周期同步，避免无效请求
  • ECharts按需渲染：仅在切换日期或账户时重新渲染图表
  • Flask Gzip压缩：API响应自动压缩，减少网络传输

🔒 安全与权限管理

API密钥管理

  • 环境变量隔离：所有API密钥存储在.env文件中，不提交到Git
  • 多账户独立凭证：4个账户使用独立的API Key/Secret/Passphrase
  • 只读权限推荐：建议使用OKX只读API密钥（仅查询持仓，不执行交易）
  • HMAC-SHA256签名：所有OKX API请求使用标准签名认证

数据访问控制

  • 文件系统权限：data/目录设置为用户私有（chmod 700）
  • Web服务鉴权：生产环境建议在Flask前配置Nginx反向代理+Basic Auth
  • CORS限制：Flask CORS配置仅允许本地或指定域名访问
  • 日志脱敏：敏感数据（API密钥、完整持仓）不记录到日志

def get_position_color(account_id, total_cost):
    A, B, C, D = get_thresholds(account_id)
    
    if total_cost == 0:
        return 'none'
    elif total_cost <= A:
        return 'green'

📊 仓位阈值配置

账户A/B（主账户、POIT）
  • A仓：初始开仓，盈利80%平仓
  • B仓：亏损-16%触发，盈利50%平仓
  • C仓：亏损-32%触发，盈利50%平仓
账户C/D（fangfang12、dadanini）
  • A仓：AB账户亏损-8%触发，盈利80%平仓
  • B仓：AB账户亏损-24%触发，盈利50%平仓
  • C仓：AB账户亏损-40%触发，盈利50%平仓
账户 仓位 触发亏损% 平仓盈利% 说明
绿色 (只有A仓)： 初始开仓，风险最低
黄色 (有AB仓)： 已补仓一次，风险中等
红色 (有ABC仓)： 已补仓两次，风险较高

📈 曲线颜色说明

  • 绿色线段：只有A仓（初始仓位）
  • 黄色线段：有AB仓（已补仓1次）
  • 红色线段：有ABC仓（已补仓2次）
  • 橙色线段：超过C阈值（风险预警⚠️）
  • 灰色线段：无仓位
  • 🟣 紫色圆点：加仓点标记（鼠标悬停查看详情）

⚙️ 开仓逻辑

  • 第一步：满足条件→开A/B账户的A仓；亏损-8%→开C/D账户的A仓
  • 第二步：再亏损-8%→开A/B账户的B仓；再亏损-8%→开C/D账户的B仓
  • 第三步：再亏损-8%→开A/B账户的C仓；再亏损-8%→开C/D账户的C仓

💰 平仓逻辑

  • A仓：上涨80%→全部平仓
  • AB仓：上涨50%→平B仓（1/2仓位）
  • ABC仓：上涨50%→平BC仓（2/3仓位）
